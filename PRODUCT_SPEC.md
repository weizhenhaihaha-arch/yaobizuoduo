# 妖币暴涨做多：V1 产品规格草案

## 1. 产品定位

这是一个面向新手的 Binance、OKX 妖币暴涨做多信号网站。

首页不承担完整行情终端的职责，只回答：

1. 现在有没有可以考虑开多的信号？
2. 这个信号为什么出现？
3. 信号现在是否仍然有效？
4. 信号消失后，之后价格实际涨了多少？

V1 是观察、提醒、历史验证和模拟评估产品，不执行真实下单。

## 2. 首页体验

### 顶部总状态

- 雷达状态：正常扫描 / 数据延迟 / 暂停扫描
- 最近更新时间
- 当前有效信号数量
- 页面主问题：`现在有可以考虑的做多信号吗？`

### 状态结果

- `可以考虑开多`：至少有一个通过策略确认的信号
- `暂时没有`：当前没有满足条件的信号
- `谨慎观察`：有候选，但数据或条件未完全确认

### 信号列表

首页不设置全局 5 个信号上限，而是分成三个区域。为了避免噪音，“没有信号”只展示进入雷达观察池的币种，不展示全市场所有币。

#### A. 已出现做多信号

- 置于最上方，绿色状态
- 标题明确写 `可以考虑开多`
- 新出现的信号排在本区域最前面
- Binance 和 OKX 同一币种分别作为独立市场记录

#### B. 潜在做多信号

- 置于中间，使用黄色或橙色状态
- 标题写 `潜在机会，等待确认`
- 说明还缺少什么条件
- 不使用“开多”措辞，避免用户误把候选当成已确认信号

#### C. 当前没有信号

- 置于最下方，使用灰色弱化显示
- 只展示币种、交易所、最近扫描时间和 `暂无条件触发`
- 默认可折叠，避免占据首屏

每个区域可以展示多个项目；过长时使用分页或“加载更多”，不静默截断。

每张卡片只放用户需要的内容：

- 币种、交易所标识、信号出现时间
- 币种前固定显示文字标识：`币安` 或 `欧易`
- 大字状态：`可以考虑开多` / `继续观察` / `信号减弱`
- 一句话触发原因
- 参考入场区域
- 信号失效条件
- 当前信号持续时间
- 信号新鲜度：最近确认时间、数据是否延迟
- 简单质量标签：`强 / 中 / 弱`，不替代信号状态
- 点击查看详情

首页不直接展示完整 OI、资金费率和多空比表格，这些放入详情页。交易所品牌色只作辅助，不能作为唯一识别方式。

同一机会在短时间内反复满足条件时，系统合并为同一个信号，不重复创建或通知；信号消失后只有重新满足完整触发条件，才允许生成新信号。

### 消失提示

信号不能从页面静默消失。信号失效时：

- 在当前列表短暂保留一条红色或灰色状态记录
- 显示消失时间和原因
- 明确写出：`停止按此信号继续埋伏`
- 仍然继续记录后续价格结果，不把信号记录删除

## 3. 信号详情页

详情页按“结论优先”排列：

1. 当前状态和用户建议
2. 触发原因，用人话解释
3. 参考入场价、失效价、信号时间
4. 简化 K 线和价格表现
5. 信号出现后的统计结果
6. 高级指标：成交量、OI、资金费率、MA/EMA 等

不使用无法解释的单一 0-100 分数作为主要结论。若未来保留评分，必须同时显示评分构成和风险提示。

## 4. 结果统计系统

每次信号生成后创建不可覆盖的信号记录，持续补充结果字段。

### 信号事实

- signal_id、策略版本、交易所、交易对
- 触发时间、可用时间、参考入场价格
- 触发原因和当时的输入快照
- 信号状态变化时间线
- 消失时间、失效原因或自然过期时间

### 结果窗口

默认研究窗口为 5 分钟、15 分钟、1 小时、4 小时、1 天；具体窗口需通过回放确认。

每个窗口记录：

- 最高涨幅
- 最大回撤
- 收盘或窗口结束涨跌幅
- 达到预设涨幅所需时间
- 是否先触发不利波动

### 统计口径

必须同时展示：

- 价格结果：信号出现后价格最高涨了多少
- 模拟策略结果：按固定入场、滑点、手续费和退出规则后的结果

不能因为价格之后曾经上涨，就直接把信号标为成功。

## 5. 前端建议

- React + TypeScript，移动端优先
- 首页使用卡片和状态标签，不做复杂交易终端布局
- 通过 REST 获取页面数据，通过 WebSocket 或 SSE 接收新信号和状态变化
- 统一组件：SignalCard、SignalStatus、EntryPanel、InvalidationPanel、OutcomeSummary、RadarStatus
- 统一状态颜色和文案，颜色不能作为唯一判断依据
- 首页首屏优先显示状态、信号和更新时间，图表延后加载

## 6. 后端建议

推荐 Python FastAPI 模块化服务，先保持一个仓库和清晰的进程边界：

1. `collector`：分别接入 Binance、OKX 的公开行情与衍生品数据
2. `normalizer`：统一交易对、时间戳、价格、成交量和数据质量状态
3. `scanner`：计算候选条件和策略状态，不直接发送用户通知
4. `lifecycle`：管理信号出现、有效、减弱、失效、过期
5. `evaluator`：按固定事件时间回填后续价格和模拟结果
6. `api`：提供首页、详情、历史、统计接口
7. `notifier`：后续接入 Telegram，发送新信号和失效通知

推荐 PostgreSQL 保存信号、行情快照、状态事件和统计结果；Redis 可作为实时状态和消息队列，但不是第一版必须项。

## 7. 关键接口草案

- `GET /api/dashboard`：总状态、当前信号、最近失效信号
- `GET /api/signals/{signal_id}`：信号详情和状态时间线
- `GET /api/signals/history`：历史信号和筛选
- `GET /api/signals/{signal_id}/outcomes`：各时间窗口结果
- `GET /api/statistics/summary`：整体统计和样本数量
- `GET /api/health`：交易所数据和扫描器健康状态
- `SSE /api/events`：新信号、信号减弱、信号消失事件

## 8. V1 明确不做

- 不自动下单
- 不保存用户交易所 API 密钥
- 不承诺盈利或准确率
- 不在首页展示所有币种
- 不用信号消失直接推导反向做空
- 不在没有历史回放证据时固定最终策略阈值

## 9. 验收标准

- 新手打开首页后 5 秒内能判断当前是否存在可考虑的做多信号
- 新信号出现后自动排在列表前面
- 信号失效时页面明确提示原因，不静默删除
- 每个信号都能查看出现、持续、消失和后续结果
- 统计结果能区分最高涨幅、窗口结束结果和模拟策略结果
- Binance 或 OKX 任一数据异常时，页面明确显示数据延迟，不继续伪装成正常信号

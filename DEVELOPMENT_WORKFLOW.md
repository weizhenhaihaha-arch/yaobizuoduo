# 妖币暴涨做多开发工作流

## 1. 总原则

本项目严格按里程碑推进。未通过当前里程碑验收，不进入下一阶段；没有记录在本文件、`PRODUCT_SPEC.md`、`DESIGN.md` 或 `PROJECT_MEMORY.md` 中的需求，不直接进入代码。

本仓库只做 Binance、OKX 的妖币暴涨做多信号，不引入另一项目的做空反转代码、策略、数据模型或 Git 历史。

V1 只做观察、信号展示、历史统计、模拟评估和可选提醒，不保存用户交易所 API 密钥，不自动下单。

## 2. 固定决策顺序

所有工作必须按以下顺序：

1. 需求与边界确认
2. 数据契约与测试样本
3. 交易所行情采集
4. 信号策略与生命周期
5. 结果统计与历史回放
6. 后端 API 与前端页面
7. 通知与稳定性
8. 纸面观察与发布评审

UI 不能先于信号定义，统计不能先于入场和结果口径，前端不能用假数据冒充真实验证结果。

## 3. 里程碑

### M0：产品和策略边界冻结

目标：把“做什么”和“不做什么”写清楚。

工作：

- 确认 Binance、OKX 的市场范围，建议先限定 USDT 永续
- 确认观察池规模和币种筛选方式
- 确认 `已出现 / 潜在 / 无信号` 三个首页分组
- 确认 `可以考虑开多 / 继续观察 / 信号减弱 / 信号消失` 文案
- 确认入场参考、失效条件、重复触发冷却和重新触发规则
- 确认结果窗口和统计字段

产出：更新 `PRODUCT_SPEC.md`、`DESIGN.md`、`PROJECT_MEMORY.md`。

通过标准：产品负责人确认；策略阈值可以暂时是候选值，但入场、失效、统计口径必须明确。

禁止：写交易执行、写最终策略阈值、直接开发完整前端。

### M1：数据契约和测试样本

目标：先定义系统要接收什么数据、怎样判断数据有效。

工作：

- 定义统一交易对、交易所、时间戳和价格格式
- 定义 K 线、成交量、OI、资金费率和数据质量字段
- 定义事件时间与数据可用时间，禁止历史回放偷看未来数据
- 制作 Binance、OKX 的脱敏固定样本和异常样本
- 定义信号、状态事件和结果记录的数据结构

产出：数据模型、JSON/数据库契约、固定 fixtures、契约测试。

通过标准：同一输入样本可重复产生同一结果；缺数据、延迟数据和错序数据都有明确状态。

禁止：接真实下单、用前端临时字段替代后端契约。

### M2：只读行情采集和数据健康

目标：稳定获得 Binance、OKX 的公开市场数据。

工作：

- 分别实现两个交易所的只读适配器
- 统一交易对映射、时间戳和精度
- 保存原始事件和标准化事件
- 实现断线重连、限频、延迟检测和数据新鲜度
- 增加采集健康接口和日志

产出：collector、normalizer、健康检查、历史采集样本。

通过标准：两个交易所能持续采集；交易所异常时系统显示数据延迟，不产生正常信号。

禁止：在采集层直接判断“买入”；不因单一交易所数据异常触发信号。

### M3：暴涨做多策略和信号生命周期

目标：把候选异动转化为少量、可解释的做多信号。

工作：

- 实现候选层、潜在信号层、确认信号层
- 使用量价、趋势、OI 等输入建立可解释条件
- 先使用候选阈值，不把直觉阈值当成最终结论
- 实现状态机：观察、潜在、确认、有效、减弱、失效、过期
- 实现信号去重、冷却和重新触发
- 保存触发原因、输入快照和状态变化事件

产出：纯函数策略模块、状态机、策略单元测试、可回放输出。

通过标准：每次信号都能回答“为什么出现、何时失效、当前是否仍有效”；同一机会不会反复刷屏。

禁止：把信号消失转成做空信号；加入真实交易执行。

### M4：历史回放和结果统计

目标：验证信号出现后价格到底发生了什么。

工作：

- 固定参考入场价和可用时间
- 计算 5 分钟、15 分钟、1 小时、4 小时、1 天结果
- 记录最高涨幅、最大回撤、结束涨跌、达到目标时间
- 区分价格统计和含手续费、滑点、退出规则的模拟结果
- 按交易所、币种、策略版本和市场环境拆分统计
- 防止样本重复、未来数据和幸存者偏差

产出：回放器、结果表、统计 API 契约、可复现报告。

通过标准：固定输入和版本可复现；统计报告能说明样本数量、缺失数据和未完成窗口。

禁止：只展示胜率；用最高涨幅直接宣称盈利；未验证就发布“准确率”。

### M5：后端 API 和实时事件

目标：为前端提供稳定、简单的产品接口。

工作：

- 实现 dashboard、signal detail、history、outcomes、statistics、health API
- 提供新信号、减弱、消失和数据延迟事件
- 处理排序：已确认、潜在、无信号
- 返回交易所文字标识、信号新鲜度和质量标签
- 增加权限边界、错误格式和接口测试

产出：FastAPI 模块、数据库迁移、API 文档、接口测试。

通过标准：前端不需要自行计算策略状态；断线、空数据和延迟都能被明确表达。

### M6：新手首页和详情页

目标：让用户 5 秒内看懂现在能不能考虑开多。

工作：

- 实现顶部雷达总状态
- 实现已出现、潜在、无信号三个区域
- 每张卡片显示币安/欧易文字标识
- 实现新信号置顶、信号新鲜度、质量标签和失效提示
- 实现信号详情、历史结果和空状态
- 移动端优先，桌面端扩展

产出：React/TypeScript 页面、组件、响应式样式、浏览器测试和截图检查。

通过标准：新手无需看高级指标即可判断状态；信号消失不会静默消失；数据异常有明确提示。

禁止：用假数据掩盖后端未完成；首页堆满 OI、资金费率和复杂图表。

### M7：通知、观测和稳定性

目标：在不打扰用户的前提下提供新信号和失效提醒。

工作：

- 先实现站内事件，再评估 Telegram
- 通知只发送确认信号、重要减弱和信号消失
- 使用去重、冷却和失败重试
- 增加监控、错误日志、数据延迟告警和任务恢复
- 进行连续纸面观察

产出：通知模块、运行手册、监控指标、纸面观察报告。

通过标准：通知和网页状态一致；系统重启后不会重复刷屏；异常可定位。

### M8：发布评审

目标：决定是否进入公开观察版本。

必须检查：

- 两个交易所数据稳定
- 信号生命周期和统计口径没有未解决歧义
- 有足够历史样本和连续纸面观察
- 统计结果没有把短暂冲高误判为盈利
- 首页、移动端、空状态和数据异常均通过测试
- 无真实下单、密钥存储或未批准的交易功能
- 文档、记忆、部署和回滚步骤完整

任一关键项失败，退回对应里程碑，不通过发布。

## 4. 每次开发任务的固定流程

1. 读取 `AGENTS.md`、`PROJECT_MEMORY.md`、`DEVELOPMENT_WORKFLOW.md` 和相关规格
2. 确认当前里程碑和本次任务是否在范围内
3. 先补测试或固定样本，再修改实现
4. 保持小范围提交，不混入其他项目和无关重构
5. 运行针对性测试，再运行本里程碑验收检查
6. 更新 `PROJECT_MEMORY.md`，只记录确认事实、变更、验证和未解决项
7. 检查敏感信息、差异和工作区状态
8. 提交并推送到 `yaobizuoduo`，网络失败时明确记录，不能假报已上传

## 5. 变更控制

如果出现新需求，先记录到 `PROJECT_MEMORY.md` 的 Open items，说明影响的里程碑；未经确认不得直接改变当前路线。

以下事项必须单独评审，不得悄悄加入：真实下单、用户 API 密钥、多用户账户、自动止盈止损、杠杆管理、做空策略、增加交易所。

## 6. 当前入口

当前项目处于 M0，已完成产品方向、首页分组、基础设计和产品规格草案；下一步只进入 M0 的边界冻结和 M1 数据契约，不直接开发完整网页或交易功能。

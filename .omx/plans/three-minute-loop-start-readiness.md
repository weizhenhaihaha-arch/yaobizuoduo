# 三分钟开发审核循环：启动准备稿

状态：流程已确认；尚未收到“启动”授权。

## 启动前硬检查

1. 当前分支、HEAD、origin/main 和工作区状态。
2. 当前唯一任务仍是 `M7-T02 awaiting_review`，未被规划文件或其他提交污染。
3. 没有其他开发 AG、旧 Aquinas、Windows supervisor 或本地脚本并发修改仓库。
4. L0 任务卡只允许审核/收口当前 M7-T02，不允许 G0、新功能、真实网络或下一阶段。
5. 明确开发执行者、Codex 审核者和三分钟监控会话责任人。
6. 用户明确说“启动”后才派发。

## 每三分钟固定检查

- 检查时间：Asia/Shanghai。
- 当前任务 ID、状态和允许范围。
- 开发 AG 的真实状态：`running` / `waiting` / `completed` / `failed`。
- 开工证据：AG 状态、工作报告、候选提交或共享工作区变化，至少一项。
- Git：分支、HEAD、origin 跟踪、未提交差异和越界文件。
- 验证：已运行测试、失败、阻塞和待审核证据。
- 路线：是否仍为 Binance/OKX USDT 永续、只做多、Paper-first、无凭证无订单。
- 产品：是否仍为普通用户默认简单视图，专业信息默认隐藏。
- 下一动作：继续等待、唤醒同一 AG、进入审核、限定返修或报告阻塞。

## 停滞处理

- 单次无变化：记录证据并继续监控。
- 连续三次无进展：向同一 AG 请求 `已完成 / 正在做 / 阻塞原因`，不得派第二张卡。
- AG 空闲但任务未完成：唤醒同一 AG，只处理当前任务。
- 出现越界、未知修改或第二执行器：立即冻结循环并报告 `BLOCKED`。

## 开发完成后的主 AG 动作

1. 要求完整报告：任务、文件、决策、命令结果、风险、分支、SHA、工作区和记忆候选。
2. Codex 独立检查完整 diff 和允许范围。
3. 在干净环境运行原验收、全量回归和对抗性 probe。
4. 只给 `ACCEPTED`、`RETURNED` 或 `BLOCKED`。
5. `RETURNED` 只唤醒同一 AG 修当前卡。
6. 每个结论保存项目记忆；未完成 exact-HEAD/merged-main/finalization 链前不派下一卡。

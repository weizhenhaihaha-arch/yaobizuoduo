# M0 产品与策略边界决策提案

## 任务信息

- 任务编号：`M0-T01`
- 目标：冻结 V1 的产品边界、信号生命周期和结果统计口径
- 状态：待主 AG 审核
- 原则：先固定可解释的接口和评估口径，具体策略阈值必须通过历史回放验证后再确定

## 一、确认事实与暂定假设

### 已确认

- V1 只支持 Binance 与 OKX。
- 产品只做暴涨启动方向的做多信号，不做做空、不自动下单。
- V1 只提供公开行情、信号展示、历史结果和模拟评估。
- 首页按“已出现做多信号、潜在信号、无信号观察项”分组。
- 每个币种必须显示交易所文字标识；同一币种在两个交易所是两条独立市场记录。
- 信号消失必须有明确原因和提示，不能静默从页面删除。

### 暂定假设

- V1 先限定为 USDT 永续合约，避免现货与合约的价格、流动性和交易规则混用。
- 观察池由两个交易所分别维护，默认纳入满足流动性和数据完整性要求的 USDT 永续，不把全市场所有币强制展示在首页。
- 首页不设固定总数量上限；分组内按优先级排序，并支持折叠或加载更多。
- 统计使用信号首次确认时的可复现参考价，不使用事后最优价格。
- 所有生产阈值先以候选配置存在，不能在 M0 宣称准确率或直接冻结为最终策略参数。

## 二、M0 决策表

| 主题 | 当前提案 | 理由 | 仍需产品确认的选择 | 下游影响 |
|---|---|---|---|---|
| 市场范围 | Binance、OKX 的 USDT 永续；只读公开行情 | 统一合约语义，降低 V1 数据和 UI 复杂度 | 是否未来单独增加现货版本 | M1 交易对契约、M2 采集器、首页交易所徽章 |
| 观察池 | 每个交易所独立维护；优先高流动性、连续可采集、USDT 计价的合约；剔除刚上线、长期停牌、数据不完整或异常价差市场 | 减少噪声和不可交易信号 | 初始池是否由固定白名单开始，还是按规则动态生成 | M1 需要池成员字段和淘汰原因；M2 需要交易对元数据 |
| 首页展示 | 三组：确认做多、潜在做多、暂无信号；无固定 5 条上限，长列表折叠/加载更多 | 让用户先看可行动项，同时保留观察范围 | 首屏默认展开多少条只影响 UI，不影响信号引擎 | M5 排序字段、M6 组件和分页行为 |
| 信号确认 | 仅在候选条件完成且数据健康时进入“可以考虑开多” | 避免把单一异动误报为入场信号 | 最终条件组合在 M3 回放后决定 | M1 字段、M3 状态机、M5 文案 |
| 入场参考 | 使用信号首次进入确认状态时的参考价，并记录时间、交易所、合约、价格来源和数据新鲜度；展示为参考区域而非保证成交价 | 保证统计可复现，避免事后挑价 | 区域宽度、滑点和手续费模型在 M4 通过回放确定 | M1 事件模型、M4 评估器、M6 入场面板 |
| 失效规则 | 数据健康前提下，价格回到启动结构下方、核心条件持续不满足、或数据超过允许延迟时，进入“信号减弱/信号消失”；原因必须记录 | 用户需要知道何时停止埋伏，也要区分市场失效和数据故障 | 具体价格结构与持续时间阈值由 M3 回放确定 | M3 状态机、M5 事件、M6 红色失效提示 |
| 重复触发 | 同一交易所+合约在一次机会结束后进入冷却；冷却内不重复通知；只有重新满足完整触发条件且形成新结构才允许重触发 | 防止出现—消失—出现的刷屏 | 冷却分钟数和“新结构”定义由 M3/M4 验证 | M3 去重键、事件幂等、后续通知 |
| 结果窗口 | 固定记录信号确认后的 5 分钟、15 分钟、1 小时、4 小时、1 天；同时记录窗口内最高涨幅、最大回撤、达到目标所需时间和窗口是否完整 | 同时衡量短线反应和延迟结果 | 是否增加更长窗口不影响 V1 核心，可由结果样本决定 | M1 结果契约、M4 回放与统计 API |
| 结果口径 | 分开显示“价格曾达到的最高涨幅”和“固定入场/退出规则的模拟结果”；样本未完成或数据缺失不计入完整样本 | 防止把短暂冲高误报为成功 | 目标涨幅、退出规则、手续费/滑点模型需回放后冻结 | M4 统计、M6 结果文案、M8 发布评审 |
| 数据异常 | 单交易所延迟、断线或字段不完整时显示数据异常，不生成正常确认信号；另一交易所可独立运行 | 防止坏数据触发错误信号 | 允许延迟阈值由 M1 数据契约定义 | M2 健康检查、M5 health API、M6 stale 状态 |

## 三、信号状态与用户文案

### 建议状态机

```text
观察中 -> 潜在机会 -> 可以考虑开多 -> 继续观察
                         |             |
                         v             v
                    信号减弱 ------> 信号消失
                         |
                         v
                       已过期
```

| 内部状态 | 首页文案 | 用户理解 | 是否可按策略考虑入场 |
|---|---|---|---|
| `watch` | 暂无信号 | 在观察池中，但没有满足条件 | 否 |
| `potential` | 潜在机会，等待确认 | 有异动，还缺少确认条件 | 否 |
| `armed` | 可以考虑开多 | 策略已完成确认，显示参考入场区域和失效条件 | 是，需遵守用户自己的风控计划 |
| `active` | 信号有效，继续观察 | 确认后的机会仍未失效 | 不代表追高；按既定计划管理 |
| `weakening` | 信号减弱，注意风险 | 条件变差，不能当作新入场信号 | 否 |
| `invalidated` | 信号消失 | 失效条件已满足，应停止按原信号埋伏 | 否 |
| `expired` | 信号已过期 | 超过观察时限或结果窗口结束 | 否 |
| `data_stale` | 数据延迟，暂不判断 | 数据不够新，暂时禁用行动建议 | 否 |

确认信号卡片最少显示：交易所、合约、状态、信号出现时间、最近确认时间、参考入场区域、失效条件、数据新鲜度和一句话原因。

失效提示模板：

> `[交易所] [合约] 做多信号已消失`  
> 原因：`[可解释原因]`。请停止按原信号继续埋伏；后续结果仍会被记录。

## 四、入场参考与失效规则边界

M0 只冻结记录口径，不冻结最终数值阈值。

- 参考入场事件：第一次从 `potential` 进入 `armed` 的事件。
- 参考价格：该事件时间点的标准化市场价格；必须带数据源时间、接收时间和质量状态。
- 参考区域：由 M3 策略根据突破/回踩结构计算，不能由前端临时生成。
- 失效事件：状态机确认核心条件失效、结构被破坏、超时或数据不可用时产生。
- 失效原因：必须使用枚举加人类可读说明，不能只存一个通用“失效”。
- 数据延迟不等同于市场失效：应进入 `data_stale` 或暂停判断，避免把采集故障统计成策略失败。

需要在 M3/M4 通过历史回放确定：结构定义、价格缓冲区、持续时间、最大信号年龄、手续费、滑点和目标收益规则。

## 五、统计窗口与结果记录

每条信号建立不可变的确认快照，并按以下窗口回填：5 分钟、15 分钟、1 小时、4 小时、1 天。每个窗口记录末价涨跌、窗口内最高涨幅、窗口内最大回撤和窗口是否完整。

跨窗口还记录：最高点时间、达到预设目标的时间、失效时间、失效原因、数据缺口、策略版本、交易所和合约。

统计页面必须同时展示最高涨幅、固定规则模拟结果、完整样本数、未完成窗口数、缺失数据数和排除原因；最高涨幅不等同于收益。

## 六、未决问题及下游影响

1. 观察池初始采用固定白名单还是动态规则，影响 M1 元数据与 M2 采集范围。
2. `armed` 的具体量价/OI 条件与阈值，影响 M3 策略和回放样本。
3. 参考区域的具体算法、滑点和手续费，影响 M4 结果可信度。
4. 冷却时长和新结构判定，影响 M3 去重与 M7 通知频率。
5. 数据延迟阈值和断线恢复要求，影响 M1 数据契约与 M2 健康检查。
6. V1 是否加入 Telegram，按当前路线不阻塞 M0-M6，建议放到 M7 决定。

## 七、M0 结论

建议主 AG 批准以上边界，进入 M1“数据契约与测试样本”。批准不代表策略已经证明有效，也不批准真实交易、自动下单、用户 API 密钥或最终阈值。M1 应先把状态、事件、快照、数据质量和结果字段变成可测试契约。
